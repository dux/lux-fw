# Deploy knowledge base

Lessons surfaced while bringing a Lux app live with this plugin on a fresh
Debian 13 / Ruby 4.0.4 host. Most are not plugin-specific - they bite any
Ruby app deployed onto a clean Linux box.

## Bundler 4 changed how install is configured

* `--deployment` and `--without` flags were removed in Bundler 4.
  Use `bundle config set --local deployment true` (or `frozen true`) and
  `bundle config set --local without 'development test'` instead.
* `deployment=true` implies `frozen=true`. If your Gemfile resolves to
  different sources per environment (see "lgem pattern" below) frozen mode
  refuses to install. Either skip frozen or commit a lockfile that uses the
  production sources.
* Lockfile and bundler version: deploying with bundler N against a lockfile
  generated by bundler M makes the host fetch + install bundler M first.
  Works, but adds a round trip - keep them in sync when you can.

## Source sync needs files git ignores

* `Gemfile.lock` is often listed in `.gitignore`. Any rsync filter that
  honors `.gitignore` (`--filter=':- .gitignore'`) will drop it - and then
  `bundle install` on the host has no lockfile to anchor to.
  Fix: force-include with `--filter='+ Gemfile.lock'` BEFORE the gitignore
  filter (filter order matters - first match wins).
* `Gemfile.lock` is platform-specific. A lockfile generated on Mac contains
  `arm64-darwin` only; bundler refuses to install for `x86_64-linux`.
  Fix once on the dev machine: `bundle lock --add-platform x86_64-linux`.

## System packages for native gem extensions

Apt/yum/pacman base packages installed by `deploy:prepare`:

* Always: `build-essential`, `libssl-dev`, `libreadline-dev`, `zlib1g-dev`,
  `libyaml-dev`, `libffi-dev`, `pkg-config`, `git`, `curl`, `rsync`
* For sequel_pg / pg: `libpq-dev` (postgresql alone is not enough - the
  runtime library is there but headers are not)
* For nokogiri (older pinned versions that build from source):
  `libxml2-dev`, `libxslt1-dev`
* For sqlite3: `libsqlite3-dev`

Older nokogiri (1.13.x) builds from source. Newer versions ship precompiled
binaries - bumping nokogiri is usually cheaper than carrying libxml-dev.

App-specific runtime deps that surfaced and are NOT installed by prepare:

* `webp` - if the Gemfile uses `cli_check 'cwebp'` or similar shellouts
* `memcached` - if `config/initializers/cache.rb` flips Lux.cache.server
  to `:memcached` in production

These should either be added to the host out-of-band or moved into the
plugin's prepare step as opt-in extras (`--with memcache`, `--with webp`).

## Apt in non-TTY contexts hangs on prompts

* Set `DEBIAN_FRONTEND=noninteractive` and pass `-y` to apt commands run
  from scripts/ssh.
* A Gemfile that shells out to `sudo apt install foo` without `-y` will
  abort on the `Continue? [Y/n]` prompt during a deploy. Move the apt
  install to prepare instead of letting bundler do it.

## ruby-build (mise) needs a safe cwd

When `mise install ruby@X` is invoked via `sudo -u <service_user> -H bash`
from an SSH session that started in `/root`, ruby-build inherits cwd
`/root`. Its internal `pushd/popd` then tries to return to `/root` and
fails with "Permission denied" - the install fails late and confusingly.
Always `cd "$HOME"` (or `cd /tmp`) at the top of the service-user block.

## systemd does not load shell profiles

* No PATH from `~/.bashrc` or `~/.profile`. A unit that runs
  `bundle exec something` where `something` shells out to `ruby` or `puma`
  via `env ...` will fail with "env: 'ruby': No such file or directory".
  Fix: set `Environment=PATH=<ruby_bin>:/usr/local/bin:/usr/bin:/bin`
  in the unit (the plugin templates do this now).
* `EnvironmentFile=` does not run shell - no `$VAR` expansion, no quoting
  rules beyond simple KEY=VALUE. Treat the `.env` like a literal map.

## Cloudflare proxy vs Caddy automatic HTTPS

Caddy's built-in Let's Encrypt issuance uses HTTP-01 or TLS-ALPN-01 to the
origin. If the Cloudflare record is proxied (orange cloud), CF terminates
TLS and the challenge never reaches Caddy - issuance fails.

Two viable shapes:

* DNS-only (gray cloud) on the apex/subdomains that Caddy handles. Caddy
  issues its own cert directly. Simplest for first deploy. (What we did.)
* Keep CF proxied with "Full (strict)" mode - install an origin cert
  separately and configure Caddy with `tls /path/to/cert /path/to/key`,
  skipping its own ACME flow. More moving parts.

For wildcards, neither HTTP-01 nor TLS-ALPN-01 works - need DNS-01.
The plugin handles the Caddyfile side: add a `tls` block to the profile
(`dns_provider`, `api_token_env`) and the renderer emits

```
*.example.com {
    tls {
        dns cloudflare {env.CLOUDFLARE_API_TOKEN}
    }
    reverse_proxy 127.0.0.1:3100
}
```

The token is written to `/etc/caddy/caddy.env` (root:caddy 0640) and
loaded via a systemd drop-in (`/etc/systemd/system/caddy.service.d/lux.conf`
with `EnvironmentFile=/etc/caddy/caddy.env`). One-time daemon-reload +
restart on first install; later deploys just reload Caddy.

The host's Caddy binary still has to include the DNS provider module,
which the preflight check enforces. Stock packaged Caddy ships without
DNS providers; build with xcaddy:

```sh
sudo apt-get install -y xcaddy
sudo xcaddy build --with github.com/caddy-dns/cloudflare \
  --output /usr/bin/caddy
sudo systemctl restart caddy
```

## Gemfiles that resolve per environment ("lgem pattern")

Common pattern in dux/* projects:

```ruby
def lgem name, github: nil
  folder = ENV.fetch('LUX_GEMS') { './.gems' }
  local  = File.join(folder, name)
  if File.directory?(local)
    gem name, path: local
  elsif github
    gem name, github: github
  else
    gem name
  end
end
```

Locally the Gemfile resolves to path: gems; on the server it resolves to
github. Two consequences:

* The local Gemfile.lock embeds local paths. Shipping it as-is + frozen
  mode -> bundler refuses to update sources. Workaround: skip frozen so
  the server can regenerate the lockfile in its own release dir.
* Uncommitted edits in any local lgem gem will silently fall away on the
  server (which pulls github HEAD instead). Before a first deploy, run
  `git status` in every local gem and push anything load-bearing.

## The plugin currently swallows remote stdout

`Open3.capture3` buffers everything until the SSH command exits. Long
steps (apt install, mise compile of Ruby, bundle install of 100+ gems)
appear silent and stuck. When they fail, the captured stderr only shows
up in the final error block - you cannot tell at a glance whether a
30-minute step is making progress or wedged.

Mitigations until streaming is wired in:

* `lux deploy:tail --app <name>` for the systemd journal after boot
* `ssh <host> 'tail -F /var/log/lux-deploy/deploy.log'` in another pane
* `journalctl -fu lux-web-<app>` while a deploy is healthchecking

## Apps live under the service user's home

Root is fixed at `/home/<service_user>/lux-apps/<app>/` and not
configurable in deploy.json. Caddy (running as user `caddy`) follows the
`/etc/caddy/sites/<app>.caddy` symlink into this tree, so the home
directory itself needs `x` for others. `ensure_remote_layout!` chmods it
to `0711` once - traversable but not listable, so other users on the box
can't enumerate `/home/<service_user>/`.

If you ever pick a service_user whose home is not `/home/<user>` (some
distros use `/var/lib/<user>` for system accounts), the derived path
won't exist and `ensure_remote_layout!` won't create it. Stick with the
`adduser`/`useradd` default to keep this simple.

## Things that are NOT the plugin's job (today)

* Host-level packages beyond Ruby/Postgres/Caddy basics
* Secrets - the plugin reads `true`-valued env keys from the caller's
  shell environment; managing those is on you
* DNS cutover - flip records yourself (we used flarectl for Cloudflare)
* Database content / seeding beyond `db:am`
