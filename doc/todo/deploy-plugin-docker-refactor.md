# Deploy plugin Docker refactor

Goal: make `plugins/deploy` boring, Docker-only, and easy to inspect on the
server. No release directories, no Ruby install on the host, no systemd units per
app. The host runs Docker Compose projects and Caddy routes domains to the
published localhost ports.

North star:

* Deploy and run Docker Compose.
* Use Caddy as the reverse proxy.
* Done.

Everything else in this design exists only to make those three things safe and
repeatable.

## Final plan

1. Keep app Docker files in `config/docker`.
2. Build full app images locally with `lux deploy:build`.
3. Save images to `tmp/deploy/<app>/images.tar.gz`.
4. Test the saved images locally with `lux deploy:test`.
5. Deploy by uploading the image archive, running `docker load`, syncing
   `config/docker`, writing `.env`, and running `docker compose up -d
   --no-build`.
6. Generate one app-root `Caddyfile` and symlink it into `/etc/caddy/sites`.
7. Use `deploy:staging` for PR/staging stacks that include their own DB in
   `compose.staging.yml`.
8. Destroy PR/staging stacks with `deploy:remove --purge --volumes`.

## Direction

This sounds like the right simplification:

* Docker is the only runtime contract.
* The deploy artifact is a full Docker image with the full app inside.
* The deploy plugin should not know how to boot Ruby, Node, jobs, or sockets.
  Docker Compose owns that.
* The deploy plugin should only build/test images locally, move or pull images
  on the server, copy/update config, run `docker compose up -d --no-build
  --remove-orphans`, write Caddy config, and reload Caddy.
* The app should keep its local Docker files under `config/docker/`.
* The remote server should have one obvious app directory with generated config,
  Docker config, logs, tmp, and a Caddyfile.
* Caddy config should live in the app directory and be symlinked into
  `/etc/caddy/sites/`.
* Remote `.env` config stays useful and should remain part of the plugin.
* Multiple public services must be first-class: web app on `DOMAIN`, socket app
  on `socket.DOMAIN`, admin app on another domain, etc.

## Non-goals

* No Capistrano-style releases.
* No rollback command in v1. Docker images can be pinned later if rollback is
  needed.
* No app systemd units. Docker restart policies handle boot.
* No host Ruby, Bundler, Puma, Node, npm, or bun setup.
* No magic generated Dockerfile. The app owns its Docker build.
* No remote source build by default. Remote hosts run already-built images.
* No Docker abstraction layer. The plugin shells out to plain `docker compose`
  commands the operator can copy/paste.
* No parsing arbitrary Caddyfile snippets to discover ports. Config is source of
  truth.

## Host layout

Default root should move out of `/home` so Caddy can read symlink targets without
home-directory permission surprises:

```text
/srv/lux-apps/
  myapp/
    manifest.json                 # resolved deploy state, no secrets
    Caddyfile                     # generated by lux deploy
    config/
      docker/                     # copied from local config/docker/
        compose.yml
        compose.prod.yml          # optional
        Dockerfile
        deploy.env                # compose/project env, no secrets if possible
    shared/
      .env                        # runtime env for containers, 0600
      log/                        # bind-mounted into containers
      tmp/                        # bind-mounted into containers
```

Caddy sees:

```text
/etc/caddy/Caddyfile              # imports /etc/caddy/sites/*.caddy
/etc/caddy/sites/myapp.caddy -> /srv/lux-apps/myapp/Caddyfile
```

Prepare must ensure the app root and Caddyfile target are readable by the caddy
process while secrets remain private:

* `/srv/lux-apps` owned by `service_user`, mode `0755`
* `/srv/lux-apps/<app>` owned by `service_user`, mode `0755`
* `/srv/lux-apps/<app>/config/docker` owned by `service_user`, mode `0755`
* `/srv/lux-apps/<app>/shared/.env` owned by `service_user`, mode `0600`
* `/etc/caddy/sites` exists and imports from main Caddyfile

Rule: all Docker-owned files live under `config/docker` locally and remotely.
Caddy is not Docker-owned, so the generated `Caddyfile` stays at the app root
where it can be symlinked into `/etc/caddy/sites`.

## Local app layout

Recommended app files:

```text
config/
  deploy.json
  docker/
    compose.yml
    Dockerfile
    compose.prod.yml              # optional
```

`config/docker/compose.yml` should work locally and remotely. It should define
`image:` names for deployment and can also define `build:` so `deploy:test` can
prebuild the full app image locally:

```yaml
services:
  web:
    image: ${WEB_IMAGE:-myapp-web:local}
    build:
      context: ${LUX_SOURCE_DIR:-../..}
      dockerfile: config/docker/Dockerfile
    restart: unless-stopped
    env_file:
      - ${LUX_RUNTIME_ENV_FILE:-../../.env}
    ports:
      - "127.0.0.1:${WEB_PORT:-3000}:3000"
    volumes:
      - "${LUX_LOG_DIR:-../../log}:/app/log"
      - "${LUX_TMP_DIR:-../../tmp}:/app/tmp"

  socket:
    image: ${SOCKET_IMAGE:-myapp-socket:local}
    build:
      context: ${LUX_SOURCE_DIR:-../..}
      dockerfile: config/docker/Dockerfile
    command: ["bun", "run", "socket"]
    restart: unless-stopped
    env_file:
      - ${LUX_RUNTIME_ENV_FILE:-../../.env}
    ports:
      - "127.0.0.1:${SOCKET_PORT:-3001}:3000"
    volumes:
      - "${LUX_LOG_DIR:-../../log}:/app/log"
      - "${LUX_TMP_DIR:-../../tmp}:/app/tmp"
```

The plugin writes `config/docker/deploy.env` remotely with values like:

```sh
LUX_RUNTIME_ENV_FILE=/srv/lux-apps/myapp/shared/.env
LUX_LOG_DIR=/srv/lux-apps/myapp/shared/log
LUX_TMP_DIR=/srv/lux-apps/myapp/shared/tmp
WEB_IMAGE=myapp-web:2026-05-17-abc123
SOCKET_IMAGE=myapp-socket:2026-05-17-abc123
WEB_PORT=3100
SOCKET_PORT=3101
```

Then it runs:

```sh
docker compose \
  --project-name lux-myapp \
  --env-file /srv/lux-apps/myapp/config/docker/deploy.env \
  -f /srv/lux-apps/myapp/config/docker/compose.yml \
  up -d --no-build --remove-orphans
```

## Image transport

Default: no registry. Build full app images locally, package them, upload them,
and load them on the server. This is the free "zip deploy" path for Docker.

Flow:

1. `lux deploy:build` builds full app images locally.
2. `lux deploy:build` writes `tmp/deploy/<app>/images.tar.gz` with
   `docker save ... | gzip`.
3. `lux deploy:test` runs those images locally and health-checks them.
4. `lux deploy` uploads the image archive to the host.
5. The host runs `docker load`.
6. The host runs `docker compose up -d --no-build --remove-orphans`.

That means the server never needs app source to build containers. It only needs
Docker, Compose, Caddy, `config/docker`, `.env`, and the loaded images.

Example:

```sh
docker save myapp-web:TAG myapp-socket:TAG | gzip > tmp/myapp-images.tar.gz
scp tmp/myapp-images.tar.gz root@srv:/srv/lux-apps/myapp/config/docker/images.tar.gz
ssh root@srv 'gunzip -c /srv/lux-apps/myapp/config/docker/images.tar.gz | docker load'
```

This is intentionally simple. The tradeoff is that it may upload full image
layers each deploy unless we add caching later.

Optional later path: registry pull.

A registry can be faster and more Docker-native because layers are cached, but
it can involve accounts, private image limits, or paid plans. Do not require it
for v1. If added, make it explicit:

```sh
lux deploy --transport registry
```

Registry flow:

1. `lux deploy:build --push` builds and pushes images.
2. `lux deploy:test` health-checks the pushed/tagged images locally.
3. `lux deploy --transport registry` copies deploy config to the host.
4. The host runs `docker compose pull`.
5. The host runs `docker compose up -d --no-build --remove-orphans`.

Recommendation for v1:

* default path: image archive with `docker save | gzip` and `docker load`
* optional later path: registry pull

## Deploy config shape

Keep `config/deploy.json`, but make the Docker shape explicit and small:

```json
{
  "default": {
    "host": "root@srv.example.com",
    "service_user": "deployer",
    "app": "myapp",
    "root": "/srv/lux-apps",
    "compose": [
      "config/docker/compose.yml"
    ],
    "image_tag": "2026-05-17-abc123",
    "images": {
      "web": "myapp-web:{{image_tag}}",
      "socket": "myapp-socket:{{image_tag}}"
    },
    "env": {
      "RACK_ENV": "production",
      "DB_MAIN": "postgres://postgres:postgres@db:5432/myapp",
      "SECRET_KEY_BASE": true
    },
    "services": {
      "web": {
        "compose_service": "web",
        "host_port": 3100,
        "port_range": [3100, 3199],
        "container_port": 3000,
        "domains": [
          "example.com",
          "www.example.com"
        ],
        "healthcheck": {
          "path": "/",
          "expect_status": [200, 301, 302]
        }
      },
      "socket": {
        "compose_service": "socket",
        "host_port": 3101,
        "container_port": 3000,
        "domains": [
          "socket.example.com"
        ],
        "healthcheck": {
          "path": "/health",
          "expect_status": [200]
        }
      }
    }
  }
}
```

Notes:

* `compose` paths are local paths copied to the remote app root.
* `images` maps logical service keys to prebuilt image names.
* `services.*.compose_service` must exist in the Compose config.
* `host_port` is the localhost port Caddy uses.
* `host_port: null` means allocate the first free port from `port_range`.
* `container_port` is documentation and validation. Compose still owns the real
  port mapping.
* `domains` can be one or many domains per service.
* `{{image_tag}}` should resolve from config or a CLI flag. A git SHA or CI build
  id is better than `latest`.
* `env` keeps current behavior:
  * string writes literal value
  * `true` means required from local ENV at deploy time
  * `false` or `null` means optional
  * `"$generate"` means generate a stable secret if missing in remote
    `shared/.env`, then reuse it on later deploys
  * `{{app}}`, `{{profile}}`, `{{config.foo}}` interpolation still works
  * `{{env.KEY}}` is allowed inside env values after required/generated env
    values are resolved

For production deploys, explicit `host_port` is easier to reason about. For
staging and PR deploys, `host_port: null` plus `port_range` keeps the command
short.

## Caddy generation

For each public service, generate one Caddy site block per domain group:

```caddyfile
example.com, www.example.com {
    @blocked {
        path *.php *.php5 *.phtml *.asp *.aspx *.jsp *.cgi
        path /.env /.git /.git/* /wp-admin /wp-admin/* /wp-login.php
    }
    respond @blocked 404

    reverse_proxy 127.0.0.1:3100
}

socket.example.com {
    reverse_proxy 127.0.0.1:3101
}
```

The generated file is written to:

```text
/srv/lux-apps/myapp/Caddyfile
```

Then symlinked:

```sh
ln -sfn /srv/lux-apps/myapp/Caddyfile /etc/caddy/sites/myapp.caddy
caddy validate --config /etc/caddy/Caddyfile
systemctl reload caddy
```

Validation should fail before reload if:

* two services in the same deploy use the same domain
* the same domain already exists in another app manifest
* two apps use the same host port
* a configured compose service is missing
* the generated Caddyfile does not validate

## Commands

Rewrite the deploy command family around the Docker flow:

```sh
lux deploy:doctor [PROFILE]       # check local + remote requirements
lux deploy:build [PROFILE]        # build full Docker images locally and save archive
lux deploy:test [PROFILE]         # run archived images locally + health checks
lux deploy [PROFILE]              # upload/load archive, sync config, compose up, caddy reload
lux deploy:staging [PROFILE]      # deploy disposable staging/PR stack with DB in Compose
lux deploy:ssh [PROFILE]          # ssh into app host/path
lux deploy:logs [PROFILE]         # remote docker compose logs -f
lux deploy:remove [PROFILE]       # docker compose down + remove caddy symlink
lux deploy:compose [PROFILE] -- ps # escape hatch: pass through to docker compose
```

`deploy:doctor` should be the safe entry point. It can check whether Docker,
Compose, Caddy, SSH, app dirs, ports, and domains are ready. If host bootstrap is
missing, print exact install/fix commands instead of trying to be clever.

Optional later:

```sh
lux deploy:prepare [PROFILE]      # one-time install/check docker + caddy
```

But v1 can skip `prepare` if `doctor` is good enough.

Remove or deprecate:

* `deploy:rollback`
* `deploy:reinstall` as a systemd concept
* `deploy:list` unless manifests make it basically free
* old `deploy:log`; replace with `deploy:logs`
* Ruby version flags
* Bundler paths
* Postgres peer-auth provisioning
* systemd unit rendering

`deploy:compose` is the emergency escape hatch:

```sh
lux deploy:compose -- ps
lux deploy:compose -- logs -f web
lux deploy:compose -- restart web
```

It should run the remote `docker compose` command with the exact project name,
env file, and compose files from the manifest/config.

## Build and test flow

`deploy:build`, `deploy:test`, and `deploy` should have crisp responsibilities:

```sh
lux deploy:build              # build images and create tmp/deploy/<app>/images.tar.gz
lux deploy:test               # load/run those built images locally, no rebuild
lux deploy                    # upload existing archive and run it remotely
lux deploy --build            # build archive first if missing/stale
lux deploy:test --build       # rebuild before local test
```

`deploy:build`:

1. Resolve config and image tags.
2. Run `docker compose config`.
3. Run `docker compose build`.
4. Tag images to the configured names.
5. Run `docker save <images> | gzip > tmp/deploy/<app>/images.tar.gz`.

`deploy:test`:

1. Resolve config.
2. Use the existing `tmp/deploy/<app>/images.tar.gz`.
3. Run `docker load` locally if needed.
4. Start the test Compose project with `docker compose up -d --no-build`.
5. Health-check local ports.
6. Run `docker compose down --remove-orphans` unless `--keep` is passed.

`deploy`:

1. Refuse to deploy if the archive is missing, unless `--build` is passed.
2. Upload `tmp/deploy/<app>/images.tar.gz`.
3. Run remote `docker load`.
4. Run remote `docker compose up -d --no-build --remove-orphans`.

## Staging and PR deploys

Yes: staging/PR deploys should be a first-class case. Same server install, same
Docker/Caddy machinery, different app name, image tag, domains, and allocated
ports.

Important staging rule: staging/PR deploys get their own database by using a
staging Compose file with Postgres inside the project. Do not provision host
Postgres databases for PR deploys.

The important rule: a PR deploy is just another app namespace.

```text
/srv/lux-apps/
  myapp/        # production
  pr-123/       # PR deploy
  pr-124/       # another PR deploy
```

Each namespace has its own:

* Compose project name: `lux-<app>`
* Caddyfile: `/srv/lux-apps/<app>/Caddyfile`
* Caddy symlink: `/etc/caddy/sites/<app>.caddy`
* env file: `/srv/lux-apps/<app>/shared/.env`
* log/tmp dirs
* Docker volumes, including its own Postgres data volume
* manifest

Recommended local files:

```text
config/
  docker/
    compose.yml                   # production/runtime services
    compose.staging.yml           # adds postgres/redis/etc for staging stacks
```

`compose.staging.yml` should define project-local stateful services:

```yaml
services:
  web:
    depends_on:
      - db

  socket:
    depends_on:
      - db

  db:
    image: postgres:16
    restart: unless-stopped
    environment:
      POSTGRES_DB: ${POSTGRES_DB:-app}
      POSTGRES_USER: ${POSTGRES_USER:-app}
      POSTGRES_PASSWORD: ${POSTGRES_PASSWORD:-app}
    volumes:
      - pg-data:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U ${POSTGRES_USER:-app}"]
      interval: 5s
      timeout: 3s
      retries: 20

volumes:
  pg-data:
```

Because Compose project name is `lux-<app>`, the DB volume is automatically
namespaced, e.g. `lux-pr-123_pg-data`.

Suggested profile:

```json
{
  "pr": {
    "extends": "default",
    "app": "{{app}}",
    "compose": [
      "config/docker/compose.yml",
      "config/docker/compose.staging.yml"
    ],
    "image_tag": "{{app}}",
    "env": {
      "RACK_ENV": "staging",
      "POSTGRES_DB": "app",
      "POSTGRES_USER": "app",
      "POSTGRES_PASSWORD": "$generate",
      "DB_MAIN": "postgres://app:{{env.POSTGRES_PASSWORD}}@db:5432/app"
    },
    "services": {
      "web": {
        "compose_service": "web",
        "host_port": null,
        "port_range": [3500, 3899],
        "domains": ["{{app}}.staging.example.com"]
      },
      "socket": {
        "compose_service": "socket",
        "host_port": null,
        "port_range": [3900, 3999],
        "domains": ["socket-{{app}}.staging.example.com"]
      }
    }
  }
}
```

Command examples:

```sh
lux deploy:build pr --app pr-123 --branch feature/foo
lux deploy:test pr --app pr-123
lux deploy:staging pr --app pr-123 --branch feature/foo
lux deploy:remove pr --app pr-123 --purge --volumes
```

`deploy:staging` is a convenience wrapper around `deploy` that expects a staging
profile/compose stack. It should refuse to run if the resolved compose list does
not include a staging compose file or a `db` service, unless `--allow-no-db` is
passed.

`deploy:staging` should use the same deploy engine as `deploy`, but with extra
guards:

* resolved app cannot be the production app name
* resolved compose files must include a staging compose file
* resolved compose config must include a `db` service, unless `--allow-no-db`
* `POSTGRES_PASSWORD: "$generate"` creates a stable per-app password in
  `shared/.env`
* generated secrets are never written to `manifest.json`

The `--branch` argument is local/CI build metadata only. It can be used to choose
the Docker image tag or label the manifest, but the remote host should not clone
or build that branch.

To keep PR deploys to "few arguments":

* `--app pr-123` selects namespace, compose project, manifest, Caddy symlink, and
  default `image_tag`.
* profile domain templates derive domains from `{{app}}`.
* `host_port: null` plus `port_range` allocates free ports automatically.
* `deploy:remove --app pr-123 --purge --volumes` destroys the PR deploy and its
  private DB volume.

Port allocation must scan remote manifests and current listeners before choosing
a port. Once allocated, ports are stored in `manifest.json` so repeat deploys of
the same PR keep the same ports.

Remove behavior for staging:

```sh
docker compose down --remove-orphans --volumes
```

Only pass `--volumes` when the operator explicitly asks for it. That makes it
possible to stop/recreate staging containers without losing the staging DB, but
still makes full PR cleanup one command.

## Local Docker test

`lux deploy:test` is the local preflight for Docker. It should run before SSH
deploys and be safe to run repeatedly.

It does not contact the remote host and does not touch Caddy. It proves that the
already-built image archive can boot locally using the same Compose files deploy
will use.

Suggested flow:

1. Resolve the deploy profile.
2. Write temporary local files under `tmp/deploy-test/<app>/`:
   * `.env` from the same env schema used for remote deploy
   * `deploy.env` with local paths and test ports
   * `log/`
   * `tmp/`
3. Run `docker compose config`.
4. Load `tmp/deploy/<app>/images.tar.gz` if images are missing locally.
5. Run `docker compose up -d --no-build --remove-orphans`.
6. Run configured health checks against local test ports.
7. Run `docker compose down --remove-orphans` unless `--keep` is passed.

Command shape:

```sh
lux deploy:test
lux deploy:test production
lux deploy:test --keep
lux deploy:test --build
```

`--build` runs `deploy:build` first, then starts and health-checks the freshly
built images.

Optional flags for image publishing later:

```sh
lux deploy:build --push
lux deploy:build --tag 2026-05-17-abc123
```

`--push` should run `docker compose push` after a successful build. This keeps
deploy itself stupid: it pulls already-tested images.

## Deploy flow

`lux deploy` should do the same boring steps every time:

1. Resolve and validate config.
2. SSH preflight:
   * Docker installed
   * `docker compose version` works
   * Caddy active
   * `/srv/lux-apps` writable by `service_user`
   * `/etc/caddy/sites` exists
3. Create remote app dirs:
   * app root
   * `config/docker`
   * `shared/log`
   * `shared/tmp`
4. Copy local `config/docker/` into remote app `config/docker/`.
5. Write `shared/.env` from the `env` schema.
6. Write `config/docker/deploy.env` from resolved deploy state.
7. Move images to the host:
   * default archive path: upload tarball and `docker load`
   * optional registry path: `docker compose pull`
8. Run `docker compose config` with the same args deploy will use.
9. Validate configured compose services exist.
10. Render app-root `Caddyfile`.
11. `caddy validate`.
12. Run `docker compose up -d --no-build --remove-orphans`.
13. Run HTTP health checks for each service that declares one.
14. Symlink Caddyfile into `/etc/caddy/sites/`.
15. Reload Caddy.
16. Write `manifest.json`.
17. Append deploy log line and print summary.

Open question: Caddy reload before or after health check. Safer v1:

* Run Compose first.
* Health-check via `http://127.0.0.1:<host_port>`.
* Only then symlink and reload Caddy.

This avoids routing public traffic to a broken container.

## Remove flow

`lux deploy:remove --app myapp`:

1. Read manifest from `/srv/lux-apps/myapp/manifest.json`.
2. Run `docker compose down --remove-orphans`.
3. Remove `/etc/caddy/sites/myapp.caddy`.
4. Reload Caddy.
5. Keep or delete app dir based on flag:
   * default: keep app dir, logs, tmp, env
   * `--purge`: delete app dir

No database removal in Docker v1 unless the database is also a compose service
and the operator passes `--volumes`.

## Doctor checks

`lux deploy:doctor` should read manifests and compare desired state to reality:

* app dir exists
* manifest parses
* compose files exist
* `docker compose ps` has expected services
* expected services are running
* localhost host ports respond
* generated Caddyfile exists in app root
* `/etc/caddy/sites/<app>.caddy` symlink points to app-root Caddyfile
* Caddy imports site files
* Caddy validates
* domain and port collisions are absent
* `shared/.env` is `0600`
* `shared/log` and `shared/tmp` exist

`manifest.json` should be enough for `list`, `doctor`, `remove`, and future
`sync-config` without local config:

```json

  "host": "root@srv.example.com",
  "service_user": "deployer",
  "root": "/srv/lux-apps",
  "path": "/srv/lux-apps/myapp",
  "compose_project": "lux-myapp",
  "compose_files": [
    "/srv/lux-apps/myapp/config/docker/compose.yml"
  ],
  "staging": false,
  "image_archive": "/srv/lux-apps/myapp/config/docker/images.tar.gz",
  "images": {
    "web": "myapp-web:2026-05-17-abc123",
    "socket": "myapp-socket:2026-05-17-abc123"
  },
  "caddyfile": "/srv/lux-apps/myapp/Caddyfile",
  "caddy_site": "/etc/caddy/sites/myapp.caddy",
  "services": {
    "web": {
      "compose_service": "web",
      "host_port": 3100,
      "domains": ["example.com", "www.example.com"]
    },
    "socket": {
      "compose_service": "socket",
      "host_port": 3101,
      "domains": ["socket.example.com"]
    }
  },
  "env_schema": {
    "RACK_ENV": "literal",
    "DB_MAIN": "literal",
    "SECRET_KEY_BASE": "required"
  },
```

For staging manifests, include `"staging": true` and known named volumes, e.g.

what will be affected. Never store resolved secret values in the manifest.

   * `deploy:staging`

## Example: main Ruby app plus socket server

Config:

```json
{
  "default": {
    "host": "root@srv.example.com",
    "app": "acme",
    "compose": ["config/docker/compose.yml"],
    "image_tag": "abc123",
    "images": {
      "web": "acme-web:{{image_tag}}",
      "socket": "acme-socket:{{image_tag}}"
    },
    "env": {
      "RACK_ENV": "production",
      "SECRET_KEY_BASE": true
    },
    "services": {
      "web": {
        "compose_service": "web",
        "host_port": 3200,
        "container_port": 3000,
        "domains": ["acme.com", "www.acme.com"]
      },
      "socket": {
        "compose_service": "socket",
        "host_port": 3201,
        "container_port": 3000,
        "domains": ["socket.acme.com"]
      }
    }
  }
}
```

Generated Caddy:

```caddyfile
acme.com, www.acme.com {
    reverse_proxy 127.0.0.1:3200
}

socket.acme.com {
    reverse_proxy 127.0.0.1:3201
}
```

This keeps the config readable: Compose defines processes, deploy config defines
public routing, Caddy is generated, and the server has one obvious app folder.
